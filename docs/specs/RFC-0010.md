# RFC-0010: Failure Recovery and Progressive Persistence

**RFC**: 0010
**Title**: Failure Recovery and Progressive Persistence
**Status**: Draft
**Created**: 2026-03-18
**Updated**: 2026-03-18
**Related**: RFC-0001, RFC-0002, RFC-0007, RFC-0009

## Abstract

This RFC introduces a failure recovery mechanism for the Soothe runner that
enables resumption of interrupted step loops and goal DAGs after a crash.
It adds progressive persistence (checkpointing after each step and goal
completion), goal DAG state restoration on resume, and a cross-validated
synthesized report for the root goal.  The design leverages the existing
but unused `DurabilityProtocol.load_state()` and
`GoalEngine.restore_from_snapshot()` APIs.

## Motivation

### Problem: All In-Flight State Is Lost on Crash

RFC-0009 introduced DAG-based step and goal execution.  However, the
runner only calls `save_state` once -- at the very end of execution.
If the process crashes mid-plan or mid-goal-DAG, all progress is lost:

| Data | Persisted? | Restored on Resume? |
|------|-----------|---------------------|
| Thread metadata | Yes | Yes (via `resume_thread`) |
| Durability state (goals, last_query) | Yes (at end only) | **No** (`load_state` never called) |
| GoalEngine (goal DAG, status, depends_on) | Yes (at end only) | **No** (`restore_from_snapshot` never called) |
| Plan / PlanStep results | **No** | No |
| StepScheduler progress | **No** | No |
| Context ledger | Yes | Yes (via `context.restore`) |
| LangGraph checkpoints | Yes (PostgreSQL) | Yes (via thread_id) |

The `DurabilityProtocol.load_state()` and `GoalEngine.restore_from_snapshot()`
APIs exist in the codebase but are never invoked.  This means goals saved at
end-of-run can never be restored.

### Problem: No Cross-Validation in Final Report

When a goal DAG completes, the `GoalReport.summary` is simply
`response_text[:500]` -- a truncated concatenation of step results.
There is no:

- Synthesis of findings across steps
- Cross-validation between parallel step results
- Consistency checking between dependent goal outputs
- Reflection-informed assessment in the final report

### Design Goals

1. **Progressive persistence**: Save state after each step and goal completion.
2. **Goal DAG recovery**: Restore goal engine state on thread resume.
3. **Plan recovery**: Restore plan and skip completed steps on resume.
4. **Daemon restart recovery**: Detect and offer to resume incomplete threads.
5. **Cross-validated final report**: LLM-synthesized summary with cross-validation.
6. **Backward compatible**: Recovery is opt-in; existing single-pass behavior unchanged.

## Architecture

### Persistence Points

```
Non-Autonomous:
  pre_stream
    |
    step_loop:
      for each batch:
        for each step:
          _execute_step()
          save_state(plan, completed_steps)  ←── NEW: after each step
        |
    post_stream
      save_state(final)  ←── existing

Autonomous:
  pre_stream
    |
    goal_loop:
      for each goal batch:
        for each goal:
          plan_creation
          step_loop (same as above with per-step saves)
          reflection
          save_state(goals, plan)  ←── NEW: after each goal
        |
    save_state(final)  ←── existing
```

### State Envelope

All progressive saves use a single structured envelope stored via
`DurabilityProtocol.save_state()`:

```python
class CheckpointEnvelope(BaseModel):
    """Progressive checkpoint for crash recovery (RFC-0010)."""

    version: int = 1
    timestamp: str
    mode: Literal["single_pass", "autonomous"]

    # Query
    last_query: str
    thread_id: str

    # Goal DAG state (autonomous mode)
    goals: list[dict[str, Any]] = Field(default_factory=list)
    active_goal_id: str | None = None

    # Plan state (current goal's plan)
    plan: dict[str, Any] | None = None
    completed_step_ids: list[str] = Field(default_factory=list)

    # Iteration tracking (autonomous)
    total_iterations: int = 0

    # Status
    status: Literal["in_progress", "completed", "failed"] = "in_progress"
```

### Recovery Flow

```
Thread Resume
    |
    v
_pre_stream:
  1. resume_thread(thread_id)           ←── existing
  2. context.restore(thread_id)          ←── existing
  3. envelope = load_state(thread_id)    ←── NEW
  4. if envelope and envelope.status == "in_progress":
       restore_goals(envelope.goals)     ←── NEW
       restore_plan(envelope.plan)       ←── NEW
       skip completed steps              ←── NEW
       emit soothe.recovery.resumed      ←── NEW
    |
    v
Step Loop:
  - StepScheduler sees completed steps
  - ready_steps() skips them
  - Execution continues from first pending step
```

### Recovery Scenarios

**Scenario 1: Crash mid-step-loop (non-autonomous)**

```
Before crash:
  Step 1: completed (saved)
  Step 2: completed (saved)
  Step 3: in_progress (not saved -- LangGraph checkpoint may exist)
  Step 4: pending

After resume:
  Step 1: completed (restored from checkpoint)
  Step 2: completed (restored from checkpoint)
  Step 3: pending (re-executed -- safe because LangGraph state is separate)
  Step 4: pending
```

Steps 1-2 results are recovered from the checkpoint envelope.  Step 3 is
re-executed because its result was not saved before the crash.  This is
safe because each step is an independent LangGraph invocation.

**Scenario 2: Crash mid-goal-DAG (autonomous)**

```
Before crash:
  Goal A: completed (saved with GoalReport)
  Goal B: in_progress, Step 1 completed, Step 2 in_progress
  Goal C: pending (depends on A, B)

After resume:
  Goal A: completed (restored)
  Goal B: active, Step 1 completed (restored), Step 2+ pending
  Goal C: pending (deps not yet met)
```

Goal A's report is fully preserved.  Goal B resumes from step 2.
Goal C waits for B to complete, preserving DAG ordering.

**Scenario 3: Daemon restart**

```
On daemon start:
  1. List threads with status="active"
  2. For each active thread:
     - Load checkpoint envelope
     - If envelope.status == "in_progress":
       - Log: "Incomplete thread {id}: {query}"
       - Optionally: auto-resume or wait for client request
```

### Cross-Validated Final Report

When the root goal of a DAG completes, the runner generates a synthesized
summary instead of truncating `response_text`:

```python
async def _synthesize_root_goal_report(
    self,
    goal: Goal,
    step_reports: list[StepReport],
    child_goal_reports: list[GoalReport],
) -> str:
```

The synthesis prompt includes:

1. **Goal description** -- what was asked
2. **Step results** -- all step outputs with descriptions
3. **Child goal reports** -- summaries from dependent goals
4. **Cross-validation instructions**:
   - Check for contradictions between step results
   - Identify information gaps
   - Note where parallel results confirm or conflict with each other
   - Highlight confidence level based on source agreement

**Fallback**: If the planner LLM is unavailable, use a heuristic summary
that concatenates step results with status headers.

### Enhanced Reflection

The `Reflection` model gains fields for richer feedback:

```python
class Reflection(BaseModel):
    assessment: str
    should_revise: bool
    feedback: str
    blocked_steps: list[str] = Field(default_factory=list)
    failed_details: dict[str, str] = Field(default_factory=dict)
```

`reflect()` implementations are enhanced to:

- Distinguish "directly failed" vs "blocked by dependency" steps
- Include failed step descriptions and truncated outputs in feedback
- Use `depends_on` to explain cascading failures

## Backward Compatibility

- `progressive_checkpoints=true` is the default but can be disabled.
- When disabled, behavior is identical to current (save at end only).
- Recovery is passive: `load_state` only triggers when `resume_thread` is
  called with a thread that has an `in_progress` checkpoint.
- Single-step plans (trivial queries) produce one checkpoint at completion,
  identical to current behavior.
- `CheckpointEnvelope.version` field enables future schema evolution.

## Constraints

- Recovery does NOT replay LangGraph steps; it skips completed Soothe steps.
- LangGraph's own checkpointing handles conversation state per thread_id.
- The two systems are complementary: Soothe checkpoints track plan/goal
  progress; LangGraph checkpoints track agent conversation state.
- Cross-validated synthesis requires a planner LLM; heuristic fallback is
  always available.
